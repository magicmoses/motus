2. LIGHTWEIGHT KG IMPLEMENTATIONS (FOR YOUR SCALE)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✓ RECOMMENDED: Materialized Views in Postgres
  Why:
    • Already using Supabase (Postgres)
    • Deterministic pipeline stages normalize data
    • Materialized views = fast read-heavy analytics
    • Can publish as RDF-compatible later if needed
    • No separate infrastructure cost

  Pattern:
    kg_nodes view (all node types unified)
    kg_edges view (relationships + jsonb properties)
    Refresh nightly after verifier stage completes

  Performance: Sub-100ms queries on 1000 papers + edges

VERDICT: Stay in Postgres. Simplest, fastest, cheapest.

3. NODE / EDGE DESIGN FOR SPORTS SCIENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Core Node Types:
  1. PAPER — DOI, title, abstract, journal, evidence_level
  2. SPORT — name, category, hierarchy (cycling → road)
  3. BODY REGION — name, anatomical_type, ICD-11 code
  4. BIOMARKER — name, category, unit, normal_range
  5. INTERVENTION — type, duration, protocol (training/supplement/recovery/device)
  6. OUTCOME — name, measurement_type, direction
  7. TOPIC — name, category (physiology/training/nutrition/biomechanics)

Edge Types:
  Paper → Paper     :CITES
  Paper → Sport     :STUDIES
  Paper → Region    :REPORTS_ON
  Paper → Biomarker :MEASURES
  Paper → Outcome   :REPORTS_OUTCOME
  Paper → Topic     :DISCUSSES
  Intervention → Biomarker :MODIFIES
  Sport → Sport     :SHARES_PHYSIOLOGY
  Topic → Topic     :RELATED_TO

KEY: Properties encode certainty (confidence, p_value, effect_size).

4. VISUALIZATION APPROACHES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✓ CYTOSCAPE.JS (for domain experts)
  • Built for biology networks
  • FCose layout handles 500 nodes smoothly
  • <200ms rendering for 1000 nodes

✗ AVOID: Force-directed graphs for hobbyists (chaotic)
✓ USE: Search-first UI + neighborhood rendering (2-3 hops, 25-100 nodes typical)

Pattern for 1000 Papers:
  1. Search bar → autocomplete (sports, topics, biomarkers)
  2. Focus node displayed in center
  3. N-hop neighborhood visualization
  4. Context panel with stats and filters

Never render full graph at once.

5. QUERY EFFICIENCY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GraphQL vs SPARQL:
  GraphQL wins for your stack (JS team, Postgres already deployed)
  • Learning curve: Low for JS devs
  • Performance: Solvable with DataLoader
  • Tooling: Excellent in Node.js

Query example: <50ms on 1000 papers
  papers(sport: "cycling", biomarker: "VO2max")
    → relatedPapers(relationshipType: "CITES")
    → outcomes { name, effectSize, confidence }

VERDICT: GraphQL. No SPARQL.

  Month 1: Schema design + Phase 1 implementation (materialized views)
  Month 2: GraphQL API built + tagger extended (parallel)
  Month 3: Basic UI + polishing

Result: Working KG explorer by end of Month 3. Not perfect, but functional.

Tight Timeline Issues:
  ⚠ Backfilling 1000+ papers into KG tables (data migration) is tedious
  ⚠ Testing query performance at scale (identifying slow queries, adding indexes)
  ⚠ Cytoscape.js integration has a learning curve (force-directed vs custom layouts)
  ⚠ You'd need to maintain the tagger extension in parallel

Faster (2 months, 30+ hrs/week sprinting):
  Possible, but requires:
    • Minimal backfill (just upcoming papers)
    • Basic Cytoscape rendering (no fancy stats)
    • Fewer edge types (focus on paper→paper, paper→biomarker)

My Recommendation:

  🎯 Defer cosmetic UI polish to Phase 3

OR:

  🎯 Do Phase 1 + 2 in 2-3 months
